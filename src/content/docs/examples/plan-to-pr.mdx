Use this when you have already figured out what needs to happen and want Skylence to execute it. Write the implementation plan in the issue body — as detailed or as high-level as you like — then add the label and walk away.

The issue body is passed verbatim to Opus. Be specific: name files, functions, and expected behavior. The more detail you provide, the more predictable the result.

**Trigger:** Add label `sky-implement` to issue.

**What it does:**

1. **Implement** (Opus, isolated worktree) — reads issue body as plan, executes each task, runs `make test` after each group.
2. **Test** (bash) — runs `make test` as final verification gate.
3. **Review** (Haiku routes → Sonnet agents) — adaptive review based on diff contents.
4. **Self-fix** (Opus) — fixes all CRITICAL and MAJOR findings.
5. **Create PR** (Sonnet) — opens draft pull request.

Write plan in issue body before adding label. Be specific: list files, functions, changes.

∆review-classify∆
Analyze implementation branch. Decide which review agents to run.

Run `git fetch origin && git diff origin/main...HEAD`.

Rules:
- run_error_handling: "true" if diff touches error handling or changes error returns.
- run_test_coverage: "true" if diff modifies source files (not just tests or docs).
- run_docs_impact: "true" if diff adds/removes public APIs, CLI flags, env vars, or config keys.

Output ONLY valid JSON:
{"run_error_handling": "true|false", "run_test_coverage": "true|false", "run_docs_impact": "true|false", "reasoning": "<brief>"}

Review implementation for issue #{{issue.number}}: {{issue.title}}

Run `git fetch origin && git diff origin/main...HEAD`.

Check:

2. Edge cases — what inputs or states could break this?
3. Conventions — matches adjacent code patterns?
4. Security — injection, path traversal, auth bypass?

List findings: severity (critical/major/minor), file:line, what to fix.
Conclude: LGTM, LGTM with minor fixes, or NEEDS CHANGES.
