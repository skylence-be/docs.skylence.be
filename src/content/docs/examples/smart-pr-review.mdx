Use this when a pull request needs a thorough review, or when you want a second opinion before merging. The workflow reads the diff and routes it to only the review agents that apply — a docs-only change skips the code review agent entirely; a change touching error handling gets a dedicated error-handling pass on top of the standard code review.

Findings are grouped by severity (CRITICAL / MAJOR / MINOR) and pushed as fixes directly onto the PR branch before you merge.

**Trigger:** Add label `sky-review` to pull request.

**What it does:**

1. **Classify** (Haiku) — analyzes diff, decides which review checks apply.
2. **Review agents** (Sonnet, parallel) — code review, error-handling review, test-coverage review run only when relevant.
3. **Synthesize** (Sonnet) — consolidates findings by severity.
4. **Fix** (Opus) — checks out PR branch, fixes every CRITICAL and MAJOR finding.

```
⊕meta⊕
name = "smart-pr-review"
description = "Adaptive PR review: classify → route to agents → synthesize → fix"
trigger.github.events = ["pull_request.labeled"]
trigger.github.label = "sky-review"
output_style = "terse"
⊕⊕

§classify§
model = "haiku"
§§

∆classify∆
Analyze PR #{{pull_request.number}}: {{pull_request.title}}

Run: `git fetch origin && git diff origin/{{pull_request.base.ref}}...origin/{{pull_request.head.ref}}`

Routing diagram (Mermaid):
flowchart LR
    diff[PR diff] --> classify[Classify what to check]
    classify -->|run_code_review=true| code_review[code-review node]
    classify -->|run_error_handling=true| error_handling[error-handling node]
    classify -->|run_test_coverage=true| test_coverage[test-coverage node]
    code_review --> synthesize[synthesize node]
    error_handling --> synthesize
    test_coverage --> synthesize
    synthesize --> fix[fix node]

Rules:
- run_code_review: "true" unless diff is empty or non-code only.
- run_error_handling: "true" if diff touches error handling or changes error returns.
- run_test_coverage: "true" if diff modifies source files (not just tests, docs, or config).
- complexity: "trivial" (typo/format), "small" (1-3 files), "medium" (4-10 files), "large" (10+ or architectural).

Output ONLY valid JSON:
{"run_code_review": "true|false", "run_error_handling": "true|false", "run_test_coverage": "true|false", "complexity": "trivial|small|medium|large", "reasoning": "<brief>"}
∆∆

§code-review§
model = "sonnet"
depends_on = ["classify"]
chain_from = "classify"
when = "$classify.output.run_code_review == 'true'"
§§

∆code-review∆
Review PR #{{pull_request.number}}: {{pull_request.title}}

Run `git fetch origin && git diff origin/{{pull_request.base.ref}}...origin/{{pull_request.head.ref}}`

Check:
1. Correctness — off-by-one, nil dereferences, race conditions?
2. Edge cases — what inputs or states could break this?
3. Conventions — matches adjacent code patterns?
4. Security — injection, path traversal, auth bypass?

List findings: severity (critical/major/minor), file:line, what to fix.
Conclude: LGTM, LGTM with minor fixes, or NEEDS CHANGES.
∆∆

§error-handling§
model = "sonnet"
depends_on = ["classify"]
chain_from = "classify"
when = "$classify.output.run_error_handling == 'true'"
§§

∆error-handling∆
Review error handling in PR #{{pull_request.number}}.

Run `git fetch origin && git diff origin/{{pull_request.base.ref}}...origin/{{pull_request.head.ref}}`

Check:
1. All errors handled — not silently swallowed?
2. Error messages specific enough for debugging?
3. Cleanup (defer, close, rollback) correct on all failure paths?
4. Any error leave state inconsistent or cause panic?

List findings: severity, file:line, what to fix.
∆∆

§test-coverage§
model = "sonnet"
depends_on = ["classify"]
chain_from = "classify"
when = "$classify.output.run_test_coverage == 'true'"
§§

∆test-coverage∆
Review test coverage in PR #{{pull_request.number}}.

Run `git fetch origin && git diff origin/{{pull_request.base.ref}}...origin/{{pull_request.head.ref}}`

Check:
1. New/changed code paths covered?
2. Edge cases tested (empty inputs, boundary values, error paths)?
3. Tests assert behavior, not just exercise code?

List gaps: severity, what to add/fix.
∆∆

§synthesize§
model = "sonnet"
depends_on = ["code-review", "error-handling", "test-coverage"]
trigger_rule = "one_success"
§§

∆synthesize∆
Synthesize review findings for PR #{{pull_request.number}}: {{pull_request.title}}

Code review: $code-review.output
Error handling: $error-handling.output
Test coverage: $test-coverage.output

Group by severity:
- CRITICAL: must fix before merging
- MAJOR: should fix before merging
- MINOR: non-blocking

If no issues: "LGTM — no fixes required."
∆∆

§fix§
model = "opus"
effort = "max"
depends_on = ["synthesize"]
§§

∆fix∆
Fix CRITICAL and MAJOR findings in PR #{{pull_request.number}}: {{pull_request.title}}

Findings: $synthesize.output

1. Checkout branch: `git fetch origin && git checkout {{pull_request.head.ref}}`
2. Fix CRITICAL and MAJOR. Fix MINOR if trivial.
3. Run `make test`.
4. Commit and push: `git push origin {{pull_request.head.ref}}`
5. If "LGTM — no fixes required", confirm and stop.
∆∆
```
